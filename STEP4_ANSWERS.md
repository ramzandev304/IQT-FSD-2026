# Step 4: Code Review & Architecture Task

## 1. What steps would you take to secure a web application?

I'd apply security at every layer rather than relying on one control. On the network/transport layer: enforce HTTPS everywhere with HSTS, and keep TLS certificates current. On the application layer: validate and sanitize all user input on the server (never trust the client), use parameterized queries/ORM bindings to prevent SQL injection, and escape output to prevent XSS. For authentication, use battle-tested libraries (never roll my own crypto), hash passwords with bcrypt/argon2, enforce strong password policies, and support MFA where possible. Sessions/tokens should be short-lived, stored in httpOnly+secure cookies (or properly scoped for SPAs), and rotated on privilege changes. Authorization should be checked on every request server-side, not just hidden in the UI. I'd add rate limiting and account lockout on auth endpoints to blunt brute-force attacks, and CSRF tokens for state-changing form submissions. Security headers (CSP, X-Frame-Options, X-Content-Type-Options) reduce the blast radius of any XSS that slips through. Secrets (DB credentials, API keys) belong in environment variables or a secrets manager, never in source control. Dependencies should be scanned regularly (npm audit, Dependabot/Snyk) since most real-world breaches come from known, unpatched vulnerabilities in third-party packages, not custom exploits. Finally, I'd log security-relevant events (failed logins, permission denials) for monitoring, run periodic dependency/penetration audits, and follow least-privilege for database users and cloud IAM roles.

## 2. How would you improve the performance of a slow React application?

I'd start by profiling with React DevTools Profiler to find the actual re-render bottleneck rather than guessing. Common fixes: memoize expensive computations with `useMemo`, wrap callbacks passed to children in `useCallback`, and wrap pure components in `React.memo` to stop unnecessary re-renders. I'd check for state placed too high in the tree causing wide re-render cascades, and split it closer to where it's used. For long lists, I'd add virtualization (react-window/react-virtual) instead of rendering thousands of DOM nodes. Code-splitting with `React.lazy`/dynamic imports and route-based chunking reduces the initial bundle size, and I'd check the bundle with a tool like `source-map-explorer` to catch accidental large dependencies. On the data side, I'd make sure API calls are deduped/cached (React Query or SWR) instead of refetching on every render, and paginate or lazy-load large datasets rather than fetching everything upfront. Images should be lazy-loaded and properly sized/compressed. I'd also check for unnecessary work in render (e.g., creating new objects/functions inline that break memoization) and unstable `key` props on lists that force full remounts. Finally, I'd measure with Lighthouse/Web Vitals (LCP, INP, CLS) to validate the fix actually moved the metrics that matter to users, not just "feels faster."

## 3. Explain the difference between SQL and NoSQL databases and when you would use each.

SQL (relational) databases like PostgreSQL or MySQL store data in structured tables with fixed schemas and relationships enforced via foreign keys. They support ACID transactions and powerful joins, which makes them the right choice when data is naturally relational, consistency matters (e.g., financial transactions, inventory counts), and the schema is relatively stable — this task manager app is a good example, since tasks have a clear, fixed shape. NoSQL databases (MongoDB, DynamoDB, etc.) store data as documents, key-values, wide columns, or graphs, trading strict schema and joins for flexibility and horizontal scalability. They're a better fit when the data shape varies a lot between records, the app needs to scale writes/reads across many servers, or the schema is expected to evolve quickly (e.g., product catalogs with wildly different attributes per category, activity feeds, caching layers). In practice, the decision comes down to: do I need strong consistency and relational integrity (choose SQL), or do I need schema flexibility and horizontal scale over strict consistency (choose NoSQL)? Many real systems use both — e.g., PostgreSQL for core transactional data and Redis/Mongo for caching or unstructured logs.

## 4. How would you deploy a full-stack application to AWS, Azure, or Google Cloud?

For a simple app like this Task Manager, I'd containerize the backend and frontend with Docker, then use a managed platform to avoid manually provisioning servers. On AWS: push the Docker image to ECR, run it on ECS Fargate (or Elastic Beanstalk for simplicity) behind an Application Load Balancer, use RDS for the PostgreSQL database, and serve the React build as a static site via S3 + CloudFront. On Azure: Azure App Service for the backend, Azure Database for PostgreSQL for the database, and Azure Static Web Apps for the frontend. On GCP: Cloud Run for the containerized backend (scales to zero, good for cost), Cloud SQL for PostgreSQL, and Firebase Hosting or Cloud Storage + CDN for the frontend. Across all three, I'd store secrets (DB connection strings, API keys) in the platform's secrets manager (AWS Secrets Manager, Azure Key Vault, GCP Secret Manager) rather than in code or plain env files, and set up a CI/CD pipeline (GitHub Actions) that runs tests, builds the Docker image, and deploys automatically on merge to main. I'd configure environment-specific variables (dev/staging/prod), enable HTTPS via the platform's managed certificates, and set up basic monitoring/alerting (CloudWatch, Azure Monitor, or Cloud Monitoring) so I know if the app goes down or the database runs out of connections.
